# AI Assistant ‚Äî v0.0.1

## Overview

AI Assistant is a personal assistant that exposes two entry points:

- **REST API** (`cmd/api`) ‚Äî text interface over HTTP; any client (web, mobile, curl) can send commands and receive Claude-powered responses directly, with no voice involved
- **Voice Daemon** (`cmd/daemon`) ‚Äî continuous voice loop: captures microphone audio, transcribes locally via whisper.cpp, processes with Claude, and speaks the response back

---

## Architecture

The project follows Clean Architecture with strict dependency inversion. Dependencies always point inward.

```
cmd/                        ‚Üê composition roots (entrypoints)
app/ai/
  entity/                   ‚Üê domain types (Message, Conversation, Memory, Intent)
  repository/               ‚Üê repository interfaces (ConversationRepository, MemoryRepository)
  usecase/                  ‚Üê use cases + input-port interfaces (CommandProcessor, HistoryManager, MemoryManager)
  service/                  ‚Üê domain services (IntentRouter, ContextBuilder, ActionExecutor)
  handler/
    http/                   ‚Üê HTTP adapters (CommandHandler, HistoryHandler)
    voice/                  ‚Üê voice adapters (Listener, Transcriber, Speaker) + audio interfaces
internal/
  config/                   ‚Üê environment-variable configuration
  database/                 ‚Üê pgx connection wrapper
  repository/               ‚Üê PostgreSQL implementations of app/ai/repository interfaces
  httpclient/               ‚Üê external client implementations (ClaudeClient, WhisperLocalClient, LocalTTSClient)
```

Each layer only knows about the layer immediately inside it, and only through interfaces.

## Entry Points

### REST API

```
POST /ai/command
Content-Type: application/json

{ "text": "What is the weather today?", "session_id": "user-1" }
```

```
GET /ai/history?session_id=user-1
GET /ai/history?limit=5
```

```
GET /health
```

### Voice Daemon

Runs a blocking loop:
1. Captures microphone audio via PortAudio with silence-based VAD ‚Äî stops recording once the user has been silent for `SILENCE_DURATION_MS` ms; `RECORD_SECONDS` is the hard upper limit
2. Sends audio to a local whisper.cpp server for transcription
3. Passes the transcribed text to `ProcessCommandUseCase` ‚Äî the same use case used by the REST API
4. Speaks the response using the OS built-in TTS (`say` on macOS, `espeak` on Linux)

The daemon and the REST API are independent entry points ‚Äî both call the same inner use case layer but have entirely separate lifecycles. The REST API does **not** require the voice daemon to be running.

## Setup

```bash

git clone https://github.com/KevenAbraham/ai-assistant
cd ai-assistant
go mod tidy

# 2. Configure environment
cp .env.example .env
# Edit .env with real values

# 3. Start PostgreSQL
make docker-up

# 4. Run migrations
make migrate-up

# 5. Run the API
make run-api

# 6. Run the voice daemon (optional)
```

## Voice Daemon Setup (whisper.cpp)

```bash
# macOS
brew install whisper-cpp
whisper-download-ggml-model base.en
whisper-server -m ~/.cache/whisper/ggml-base.en.bin --port 9000
```

```bash
# Linux
git clone https://github.com/ggerganov/whisper.cpp
cd whisper.cpp && make
bash models/download-ggml-model.sh base.en
./server -m models/ggml-base.en.bin --port 9000
```

Then set `WHISPER_URL=http://localhost:9000` in `.env`.

## Dependency Injection (REST API)

The REST API uses [Uber Fx](https://github.com/uber-go/fx) for dependency injection, split into four modules:

| Module | Provides |
|---|---|
| `ConfigModule` | `*config.Config` |
| `DBModule` | `*database.DB`, `ConversationRepository`, `MemoryRepository` |
| `AIModule` | `AIClient`, `CommandProcessor`, `HistoryManager`, `MemoryManager`, HTTP handlers |
| `ServerModule` | `*zap.Logger`, HTTP server lifecycle |

The voice daemon wires dependencies manually ‚Äî its linear lifecycle does not benefit from a DI framework.

---

## Key Design Decisions

**No fx in the daemon.** The daemon is a flat, sequential loop with a trivial lifecycle (connect ‚Üí loop ‚Üí close). Introducing fx would add module overhead, lifecycle hooks, and DI graph resolution with no architectural benefit. Manual wiring is clearer and more direct here.

**Interfaces at every boundary.** All cross-layer dependencies use interfaces defined in the consuming layer or in the use case ports (`usecase/ports.go`). Concrete types never escape their own layer.

**Local-only AI dependencies.** Transcription uses whisper.cpp (local, no API key). TTS uses the OS built-in engine. The only cloud dependency is the Anthropic Claude API for language understanding.
